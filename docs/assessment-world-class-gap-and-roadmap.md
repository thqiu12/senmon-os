# Compass — 世界顶尖 SaaS 差距评估与路线图

> 评估时点:2026-06(SQLite→Postgres 迁移 + 三层备份完成后)。
> 依据:本轮实际审阅的代码与运维(部署脚本 / CI / 36 模型 schema / 备份 / 迁移工具 / 审计日志 / 本次切换过程)。标注 *推断* 的项未做深审。
> 校准:面向**精益小团队 + AI 协作**,DoD 取"世界顶尖里可被小团队达成"的水位,**不镀金**(明确列出"不做什么")。

## 0. 两个镜头与总评

| 镜头 | 它是什么 | 总评 |
|---|---|---|
| **A. 单个学校法人的招生+在籍系统**(今天实际的活) | 自用的生产系统 | **≈ 7.3 / 10** —— 运维一流、功能厚实的好系统 |
| **B. 卖给全世界学校的多租户 SaaS**(目标) | 商业产品 | **≈ 3.5–4 / 10** —— 优秀的"地基阶段" |

**一句话结论:技术领先于流程。** 差距集中在 **多租户 + 发布纪律 + 安全 + 可观测**,而**不在招生功能本身**(功能已经很能打)。Postgres 迁移是必须的第一块地基石,楼还没盖。

---

## 1. 维度逐项(镜头 B / 全球 SaaS)

每项格式:**现状 → 世界顶尖 DoD → 差距 → 优先级**。

### 1.1 多租户与数据隔离 — 当前 2/10 — **P0(阻断)**
- **现状**:单租户。全表无 `organizationId`,无租户边界。现在装第二所学校的数据没有任何隔离。
- **DoD**:① 租户模型 `Organization`(→法人→校);② Prisma `$extends` 查询扩展**强制**注入 `organizationId`(应用层主隔离);③ Postgres **RLS** 作纵深防御;④ 子域名 + 学生端自定义域路由到租户;⑤ **跨租户泄漏测试**(A 租户绝对查不到 B 的任何一行)纳入 CI。
- **差距**:全部待建 = 既有 Plan 2(多租户)+ Plan 3(RLS+索引)。
- **精益校准**:行级 `tenantId` + Prisma 扩展强制 + RLS,**不要**一上来分库/分 schema。

### 1.2 发布工程 / 流程成熟度 — 当前 4/10 — **P0**
- **现状**:部署自动化**很成熟**(原子零停机、healthcheck 失败自动回滚、部署前备份)。但:**无 staging 环境;CI 不拦自动部署;agent/人直推生产分支 `chore`**。本次迁移就栽过——截断分支被推、两个 agent 抢推 prod、`.env` 带外切换。
- **DoD**:① `main`/生产分支**受保护**,禁直推;② 一切改动走 **PR + CI 绿**才能合;③ **staging 环境**镜像生产(独立 Supabase + 部署),先验后上;④ 部署可追溯、可回滚(回滚已有);⑤ agent 也只能开 PR,不能直推。
- **差距**:立分支保护 + staging + CI gate。**最便宜、见效最快、今天刚栽过。**
- **精益校准**:用 GitHub 原生分支保护 + 一个小 staging,别上重型 CD 平台。

### 1.3 安全与合规(替别校托管 PII) — 当前 4/10(*部分推断*) — **P1**
- **现状**:有审计日志、限流(`lib/rateLimits`)、CSRF/session、角色守卫(`isCoreAdmin`、`role===GUEST` 鉴权铁律)。但**无正式威胁建模 / 逐路由授权审计 / 渗透 / 依赖扫描**。
- **DoD**:① **逐 API 路由授权矩阵**审计(谁能读写什么)+ 租户隔离测试;② 密钥管理(集中、可轮换);③ PII 静态加密(Supabase 已有)+ 字段级敏感项评估;④ **个人情报保护法/GDPR**:数据导出 & 删除能力(随多租户一起做);⑤ 依赖漏洞扫描进 CI;⑥ 面向机构客户的**安全问卷答得出来**。
- **差距**:授权审计 + 租户隔离 + 导出/删除 + 依赖扫描。
- **精益校准**:**先不急 SOC2/ISMS**;把"控制项 + 能讲清楚的安全叙事"做扎实,真有大单要求再去认证。

### 1.4 可观测性与可靠性 — 当前 4/10 — **P1**
- **现状**:healthcheck(部署后 API+CSS 探活)、备份失败邮件(Resend)、Sentry 配置文件已在(`sentry.*.config.ts`)、healthcheck ping 钩子。
- **DoD**:① Sentry 告警**真正接通**(错误实时到人);② 基础指标(请求量/错误率/延迟/DB 连接);③ **uptime 外部监控 + 告警**;④ 备份**成功**的主动监控(ping 途断即告警,钩子已留);⑤ 简单 SLO + 事故 runbook(切换 runbook 已有,扩成通用)。
- **差距**:把已有的脚手架(Sentry/ping 钩子)接通 + 加一个外部 uptime。
- **精益校准**:全用托管(Sentry / UptimeRobot / Supabase 内置指标),**别自建监控栈**。

### 1.5 国际化 / 本地化 — 当前 4/10 — **P2**
- **现状**:i18n 脚手架在(`lib/i18n/en.ts`),但底子是日本;UI 时间已有 `<LocalTime>` 设备时区惯例。
- **DoD**:① 全 UI 文案走 i18n key;② locale / 时区 / 货币可配;③ 至少 ja / en / zh 三语;④ 数字/日期本地化。
- **差距**:把硬编码文案收进 i18n;补语言包。
- **精益校准**:**日本先行**,i18n 框架先立好(避免日后大返工),语言包按市场逐步加。

### 1.6 性能与可扩展 — 当前 6.5/10 — **P2**
- **现状**:当前低量(十几条)毫无压力;pgbouncer 连接池已配;列表已分页(50/页)。
- **DoD**:① 关键查询有索引(= Plan 3);② N+1 审查;③ **多租户下查询不退化**(租户过滤命中索引);④ 一条负载测试基线。
- **差距**:加索引 + N+1 审查(随多租户做)。
- **精益校准**:加索引 + 分页审查即可,**别提前分片/读写分离**。

### 1.7 数据生命周期 — 当前 7/10(已不错) — **P1(收尾)**
- **现状**:三层备份(Supabase Pro 日备 + R2 异地 pg_dump + 冻结 SQLite 快照);`migrate deploy` 非破坏纪律。
- **DoD**:① **定期"真恢复"演练**(把 R2 dump 实际 `pg_restore` 到空库验证可恢复);② 保留策略(已有 90 天);③ 个人情报**导出/删除**能力(随多租户做);④ 迁移评审(破坏性变更需双人/PR 确认)。
- **差距**:做一次恢复演练并写进日历;导出/删除接口。
- **精益校准**:恢复演练每季度一次足矣。

### 1.8 商业化 / 可售卖 — 当前 4/10 — **P2→P3**
- **现状**:内部文档强(runbook/spec/plan);但无计费 / onboarding / 对外文档 / SLA / 支持渠道。
- **DoD**:① 开租户的(半)自助 onboarding;② 按 Organization 订阅计费;③ 客户侧管理后台;④ 对外文档 + SLA + 支持入口;⑤ 定价。
- **差距**:全部待建。
- **精益校准**:**先半自助**(你手动开租户)验证 PMF,跑通几家再自动化计费,别过早建账单系统。

## 2. 优先级路线图(按业务影响)

> 原则:**不做不能卖的先做(P0)→ 托管他校数据的底线(P1)→ 规模化与体验(P2)→ 商业放大(P3)**。

- **P0 地基(不做就没有 SaaS 命题)**
  1. **多租户隔离**(Plan 2:`organizationId` + Prisma 强制扩展)
  2. **RLS + 索引**(Plan 3:纵深隔离 + 租户查询性能)
  3. **发布纪律**:分支保护 + staging + CI gate(禁 agent/人直推生产)

- **P1 托管他校数据的底线**
  4. 安全:逐路由授权审计 + 跨租户泄漏测试(进 CI)
  5. 可观测性:Sentry 告警接通 + 外部 uptime + 备份监控
  6. 数据:备份**恢复演练** + 个人情报导出/删除

- **P2 规模化与体验**
  7. 性能:索引 + N+1(随多租户)
  8. i18n 框架落地(日先行)
  9. 文件存储多租户化(Plan 4:R2 按租户路径)
  10. 半自助 onboarding(手动开租户验证 PMF)

- **P3 商业放大**
  11. 订阅计费自动化 + 客户后台
  12. 对外文档 / SLA / 支持
  13. 模块框架 + `isAdmin`→能力化(Plan 5)
  14. 多区域(全球扩张时)

---

## 3. 本周级"最便宜见效"3 件(立刻能做)

1. **GitHub 分支保护**:`main` / `chore` 禁直推,必须 PR + CI 绿才能合。≈ 1 小时,**今天刚因此栽过**。
2. **接通告警**:Sentry 告警 + UptimeRobot 监控生产 URL。≈ 1–2 小时。
3. **备份恢复演练**:把今天的 R2 `db-*.dump.gpg` 真的 `pg_restore` 到一个空 Supabase 项目,验证"备份能恢复"。≈ 1 小时(否则备份只是"看起来有")。

---

## 4. 明确不做(避免镀金,守住精益)

- 不提前分库/分片/读写分离(行级 `tenantId` 撑很久)。
- 不自建监控/日志栈(用 Sentry/UptimeRobot/Supabase)。
- 不追 SOC2/ISMS 认证(等有大单要求)。
- 不上 K8s;保持 **PM2 + VPS + Supabase** 直到规模真的逼着换。
- 不过早建计费系统(先半自助验证 PMF)。
