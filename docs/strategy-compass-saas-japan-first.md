# Compass プロダクト戦略 — Japan First → Global SaaS（2026-06-22）

## ポジショニング
- **Compass** = 専門学校・大学・日本語学校向けの「**出願 → 在籍 → 卒業**」一気通貫 SaaS。
- **chinichi_OS** = 私塾（試験なし獲客型）向け。**顧客が別**なので食い合わない。
- **護城河（差別化）**: 留学生の**在留資格/出席率コンプライアンス**（入管報告は法的必須）＋出願〜在籍の一体運用＋AI支援。日本語学校の在籍管理ともほぼ同一機能 → TAM は 専門学校＋大学＋日本語学校。

## 評価（対標 全球トップ SaaS, 10=最高）
- 単一テナント・日本向け**アプリ**としては ≈ 6.5/10（工程の底は強い）。
- **グローバル SaaS プロダクト**としては ≈ 3/10。理由: 多租户=0 / 計費=0 / SSO=0 / i18n=ほぼ無 / 単一リージョン単機(SQLite) / 合規=未着手。
- 差別化(留学生コンプラ+全流程+AI)は本物で**世界共通の需要**。差距は「既知・順序付け可能なレイヤー群」。

## GTM 戦略: Japan First
一足飛びに worldwide にしない。**日本の留学生教育で多テナント SaaS を確立 → 有料顧客と事例 → その資金/事例でグローバル化**。

## ビルド・ロードマップ（Japan First）
- **Phase 0（数日・ほぼコード不要）**: 既存の作り込みを「点火」。Sentry/監視ON、メール失敗の可視化、各校 notifyEmail、(任意)AIキー。
- **Phase 1（地基・売る前提）**: **多テナント基盤 + PostgreSQL + HA基礎 + 可観測/SLA/状態ページ + テナント開通/ブランド + 計費(Stripe)**。← これが「外部の学校に売れる」前提。最優先・本spec対象。
- **Phase 2（楔=差別化機能）**: **留学生 在籍コンプラ**(出席率→入管報告、在留期限管理)＋日本人/留学生の在籍分流。既存の在籍モデル(School/Class/Student/Attendance/…)を完成・点火。
- **Phase 3（売れる形に）**: **学生 App(Capacitor=擬似ネイティブ・1コードでiOS/Android/PWA・テナント別ブランド)** ＋ オンライン支払 ＋ セルフ開通/オンボーディング。→ 日本で対外販売開始。
- **Phase 4（グローバル化）**: i18n框架(next-intl, 時区/通貨)・合規(GDPR/FERPA/データ駐留 多区域)・SSO/SAML/SCIM・WCAG AA・SOC2/ISO 路線・開放API/LTI。

## 確定した決定
- テナント境界 = **法人**（一法人=一テナント、配下に複数校。既存 ApplySchool.hojin/多校構造にきれいに写像）。
- 計費 = **サブスク × 法人**。
- テナント隔離 = **共有DB + tenantId + Postgres 行レベルセキュリティ(RLS)**（運用最省・主流・売れる。強隔離が要る大学には将来「専用DB」企業版）。
- 学生App = **Capacitor**（純ネイティブは多テナント保守に不経済）。
- 順序 = 先に内部で在籍を点火しつつ、**学生App/対外販売の前に必ず多テナント基盤(Phase 1)を入れる**（後付けretrofitは死亡行軍）。

## 未決定（Phase 1 spec で詰める）
- 既存データのテナント写像（知日グループ1テナントに2法人=羽場/平井をぶら下げる? それとも法人ごとに別テナント?）。
- テナント識別（サブドメイン vs ログイン時選択）。
- Postgres 移行手順（ダウンタイム最小化、SQLite→PG データ移送、RLS導入）。
- 計費プラン設計（学生数/校数 課金軸）。
