# SQLite → Neon (Postgres) Migration — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Move Compass from single-file SQLite to managed Neon Postgres (Tokyo) with zero behavior change — removing the scalability ceiling and single-point-of-failure, and laying the base for multi-tenancy (Plan 2).

**Architecture:** Lift-and-shift, single-tenant. Switch Prisma provider to `postgresql`, adopt Prisma **migrations** (replacing `db push --accept-data-loss`), fix the SQLite→PG behavior gaps (chiefly case-insensitive `contains`), point tests + CI + deploy at Postgres, copy existing data with a verified script, cut over in a short maintenance window with a rollback path.

**Tech Stack:** Next.js 14, Prisma 5, PostgreSQL (Neon, ap-northeast-1 Tokyo), vitest, Playwright.

### Task 2: Fix the case-insensitive search gap (SQLite → PG)

Context: SQLite `contains` is case-insensitive by default; Postgres `contains` is **case-sensitive**. Every `{ contains: ... }` used for human search must add `mode: "insensitive"` to keep behavior identical.

**Files (audit + fix all `contains` used for search):**
- `app/api/applications/route.ts` (the `where.OR` search block: lastName/firstName/kana/applicationNo/email/schoolName)
- `app/api/admin/enrollment-list/route.ts` (`schoolName: { contains }`)
- `app/api/admin/audit-logs/route.ts` (summary/targetLabel/actorName search)
- `app/api/announcements/recipients/route.ts` (if it filters by name)
- Any other `contains` used on user-entered search (grep below)

- [ ] **Step 1: Find every `contains` to triage**

Run: `grep -rn "contains:" app/api lib | grep -v node_modules`
Action: for each that is a **human search** (names, free text), add `mode: "insensitive"`. Leave exact-key `contains` (e.g. enum-ish) as-is.

- [ ] **Step 2: Add `mode: "insensitive"` (example — applications search)**

```ts
where.OR = [
  { lastName:       { contains: search, mode: "insensitive" } },
  { firstName:      { contains: search, mode: "insensitive" } },
  { lastNameKana:   { contains: search, mode: "insensitive" } },
  { firstNameKana:  { contains: search, mode: "insensitive" } },
  { applicationNo:  { contains: search, mode: "insensitive" } },
  { email:          { contains: search, mode: "insensitive" } },
  { schoolName:     { contains: search, mode: "insensitive" } },
];
```
Apply the same `mode: "insensitive"` to the other search `contains` found in Step 1.

- [ ] **Step 3: Typecheck**

Run: `npx tsc --noEmit`
Expected: 0 errors (`mode` is valid for the postgresql provider).

- [ ] **Step 4: Commit**


### Task 4: Data migration script (SQLite → Postgres), count-verified

**Files:**
- Create: `prisma/migrate-sqlite-to-pg.ts`

- [ ] **Step 1: Write the migration script**

Two Prisma clients: source (SQLite via a saved `SQLITE_URL`) and target (PG via `DATABASE_URL`). Copy tables in FK-safe order; verify row counts; idempotent (skip if target already populated). Pseudocode of the core:

```ts
import { PrismaClient as Pg } from "@prisma/client";          // target (DATABASE_URL)
import { PrismaClient as Sqlite } from "@prisma/client";       // source (SQLITE_URL)

const src = new Sqlite({ datasources: { db: { url: process.env.SQLITE_URL! } } });
const dst = new Pg();

// FK-safe order: parents first
const ORDER = [
  "adminUser", "applySchool", "applyDepartment", "cohort", "agent",
  "application", "applicationSchool", "document", "adminNote",
  "enrollmentProcedure", "enrollmentSignature", "enrollmentQuota",
  "interviewFeedback", "auditLog", "formFieldConfig", "systemSetting",
  // school-mgmt cluster:
  "school","class","subject","teacher","student","timetable","timetableSlot",
  "attendance","leaveRequest","homework","homeworkSubmission",
  "certificateRequest","calendarEvent","schoolNotice","chatMessage",
];

for (const model of ORDER) {
  const rows = await (src as any)[model].findMany();
  if (rows.length) await (dst as any)[model].createMany({ data: rows, skipDuplicates: true });
  const got = await (dst as any)[model].count();
  console.log(`${model}: src=${rows.length} dst=${got}`);
  if (got < rows.length) throw new Error(`COUNT MISMATCH ${model}: ${rows.length} → ${got}`);
}
```
Notes: use the real Prisma model names from `prisma/schema.prisma`. `createMany` is fine on PG. Run in dependency order so FKs resolve.

- [ ] **Step 2: Dry-run against a fresh Neon `shadow`/dev branch**

Run: `SQLITE_URL="file:./prod-copy.db" DATABASE_URL="<neon-dev>" npx tsx prisma/migrate-sqlite-to-pg.ts`
