# 出願者タイプ（日本人 / 留学生）対応 — 設計仕様

- 日付: 2026-06-18
- 対象: senmon / Compass（入学・出願システム）
- ステータス: 設計確定（ユーザー承認済み）。実装はフェーズ分割。

## 1. 背景・目的

現状の Compass は外国人留学生向けの出願に最適化されており、出願フォームに在留資格・在留期限・日本語レベル等の留学生専用項目が含まれる。今後は**日本人学生にも使えるようにしたい**。

そのため:
1. 出願の最初に **「日本人 / 留学生」を選択**し、その後に志望校を選ぶ。
2. 日本人の場合は **志望校（学科）・フォーム・費用・手続き**を日本人向けに変えられる。これらは**管理画面で編集できる**。
3. 現状 **志望校管理（ApplySchool）と form管理が連動していない**（form管理の学校タブがハードコードで、志望校を追加してもフォームタブが増えない）。これを連動させる。

## 2. 用語

- **applicantType（出願者タイプ）**: `japanese`（日本人）/ `foreign`（留学生）の2値。
- 既存データ・未指定は **`foreign` を既定**とする（現状は留学生前提のため、非破壊）。

## 3. 中核アプローチ

既存の仕組みに「タイプ」という**次元を1本追加**する（日本人用フォームを別テーブルで丸ごと複製しない）。
理由: 現状の FormFieldConfig は既に「項目ごとの有効/無効・学校別(`schoolId`)」を持つため、そこに `applicantType` 次元を足すのが最小改修かつ柔軟。複製方式は重複・保守コストが高く不採用。

公開中の出願フロー（売上の入口）を触るため、**全フェーズで非破壊**（既定=留学生で従来動作を維持）を徹底する。

## 4. データモデル変更

| モデル | 追加/変更 | 備考 |
|---|---|---|
| `Application` | `applicantType String @default("foreign")` を追加 | 既存データは foreign 既定（追加のみ＝db push 安全） |
| `FormFieldConfig` | `applicantType String?` を追加（null=共通）。一意制約を `@@unique([fieldKey, schoolId])` → `@@unique([fieldKey, schoolId, applicantType])` に拡張 | 既存行は applicantType=null のまま 3列で一意性維持。SQLite の制約変更はテーブル再構築になるため要検証（既存データのバックアップは auto-deploy が取得） |
| `ApplyDepartment` | `applicantTypes String @default("both")` を追加（`japanese`/`foreign`/`both`） | Phase 2。学科単位でタイプ指定 |
| 費用/手続き | タイプ別設定の保持先（既存 SystemSetting/該当モデル）に type 次元を追加 | Phase 3。詳細は Phase 3 着手時に確定 |

## 5. 出願フロー（申請者画面 app/apply）

新しい順序:
1. **タイプ選択（新ステップ/最初の画面）**: 「日本人 / 留学生」を選ぶ。選択は出願 state に保持し、提出時に `Application.applicantType` へ保存。
2. **志望校・学科**: 学科を applicantType で出し分け（Phase 2。Phase 1 では全学科表示のまま）。
3. **フォーム**: `/api/apply/form-config?schoolId=...&type=japanese|foreign` で **(学校 × タイプ)** の設定を取得し、項目を出し分け。
   - 例: 留学生=在留資格/在留期限/日本語レベル/国籍を表示・必須、日本人=これらを非表示にし高校名等を表示。
4. **費用**: タイプ別（Phase 3。Phase 1 では従来通り）。
5. **確認・提出**。

### フォーム設定のマージ順（/api/apply/form-config）
`FORM_FIELD_DEFAULTS（タイプ既定）` → `全校共通(共通)` → `全校共通(type)` → `学校(共通)` → `学校(type)`。**type 指定が共通を上書き**。
- `?type` 未指定時は従来挙動（共通のみ）にフォールバック。
- 既定値はタイプ別に用意（japanese 既定は留学生専用項目を無効化）。

## 6. 管理画面変更

### 設定する場所（差を作る）
- **各種設定 → form管理**:
  - **学校タブをハードコードから ApplySchool（ライブ）に変更**（= 志望校管理↔form管理の連動修正。最優先）。
  - **タイプ切替（共通/日本人/留学生）**を追加。学校×タイプでフォーム項目を編集。
  - `/api/admin/form-config` GET/PUT が `schoolId` に加えて `applicantType` を受け取る。
  - 必須書類（file 項目）も同じ仕組みでタイプ別に分かれる。
- **各種設定 → 志望校管理**（Phase 2）: 学科編集に **対象タイプ（日本人/留学生/両方）** を追加。
- **費用設定・入学手続き**（Phase 3）: タイプ別。

### 見る/絞る場所（差を区別する）
- **申請詳細**: 出願者の **タイプをバッジ表示**。表示フォーム項目もその人のタイプに従う。
- **選考管理 / 申請一覧**: **タイプで絞り込み・列表示**（日本人/留学生を分けて選考）。
- **定員管理**: 学科がタイプ別（Phase 2）になるため、定員も自然に分かれる。
- **分析・予測 / ダッシュボード**: タイプ別集計（任意・後続）。

## 7. フェーズ分割と受け入れ基準

### Phase 1 — タイプ選択 + フォーム連動 + 連動修正（核）
- `Application.applicantType` 追加、出願フローに「日本人/留学生」選択ステップ。
- `FormFieldConfig.applicantType` 次元、form管理にタイプ切替、`/api/apply/form-config` と `/api/admin/form-config` の type 対応。
- **form管理の学校タブを ApplySchool 連動に修正**。
- 申請詳細にタイプ表示、申請一覧/選考管理にタイプ絞り込み。
- **受け入れ基準**: 出願者がタイプを選ぶ→学校→そのタイプ用フォームが出る。管理画面で学校×タイプのフォームを編集でき、志望校を追加するとform管理にその学校が出る。一覧でタイプ絞り込みができる。既定（type未設定/既存データ）は従来動作。

### Phase 2 — 学科のタイプ出し分け
- `ApplyDepartment.applicantTypes` 追加、志望校管理で設定、出願フローの学科フィルタ。
- **受け入れ基準**: 学科にタイプを設定でき、出願時に選んだタイプの学科だけ表示される。定員もそれに沿う。

### Phase 3 — 費用・手続きのタイプ別
- 選考料/学費・入学手続き内容をタイプ別に設定・適用。

## 8. 非破壊・移行

- すべて**追加カラム中心**（既定値あり）。`Application.applicantType` 既定=foreign、`FormFieldConfig.applicantType` 既定=null（共通）、`ApplyDepartment.applicantTypes` 既定=both。
- `/api/apply/form-config` は `type` 未指定で従来挙動。出願フローのタイプ選択を入れるまで既存出願は留学生として動作。
- `FormFieldConfig` の一意制約変更は SQLite ではテーブル再構築になり得る。auto-deploy のデプロイ前バックアップで保全しつつ、ローカルで `prisma db push` を事前検証する。
- 既存の英語対応（i18n）と独立。タイプ選択ステップの文言も `t()` で英訳辞書に追加。

## 9. スコープ外（YAGNI）

- 日本人/留学生以外のタイプ区分は作らない（2値で十分）。
- 学校ごとのフォームを丸ごと複製する方式は採らない（タイプ次元で表現）。
- 分析・ダッシュボードのタイプ別表示は任意（必要になってから）。
